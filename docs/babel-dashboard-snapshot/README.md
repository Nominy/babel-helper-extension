# Babel Dashboard Snapshot Docs

This directory documents the saved Babel Audio transcription dashboard snapshot that lives under `decompiles/`.

Important scope limits:
- This is not the original Babel source repository.
- It is a browser "Save Page" capture of one rendered page plus downloaded assets.
- Everything in these docs is either:
  - directly observed from the saved HTML/CSS/JS files, or
  - an inference from framework/runtime patterns in those files.
- If a statement is inferred rather than directly visible, it is labeled as such.

What this snapshot covers:
- A single page saved from `https://dashboard.babel.audio/transcription/RU-transcription`
- The rendered DOM for that moment in time
- The static asset set that the browser saved locally
- Enough runtime structure to build DOM-based helper extensions

What this snapshot does not cover:
- Server-side APIs
- Hidden React source code before bundling
- TypeScript types
- Build scripts
- Project structure from Babel's real repository
- Any page that was not loaded into this saved capture

Recommended reading order:
1. `dom-and-workflow.md`
2. `runtime-and-assets.md`
3. `extension-engineering.md`

High-confidence conclusions:
- The dashboard is a Next.js App Router application.
- The transcription page is built with React and hydrated client-side.
- The page uses Tailwind-style utility classes heavily.
- Several controls expose stable accessibility labels, which are better extension hooks than class names.
- Some important actions (for example delete and merge) are routed through a per-row dropdown menu rather than directly exposed as buttons.
- The app uses multiple third-party integrations: Clerk, Intercom, Google Maps, GTM, Meta Pixel, Reddit Pixel, Bing UET, LinkedIn Insight, and PostHog-related scripts.

Extension design rule for this environment:
- Prefer driving existing controls (buttons, menus, toggles, textareas) over mutating the DOM or synthetic state directly.
- Avoid coupling to generated ids like `radix-...` or long utility-class chains.
- Expect React re-renders and portal-based menus.

Glossary used in these docs:
- "Observed": directly visible in the saved files.
- "Inferred": derived from naming patterns, framework signatures, or DOM behavior, but not proven by source.
- "Stable selector": a selector likely to survive re-renders and version bumps better than ids/classes generated by the framework.
